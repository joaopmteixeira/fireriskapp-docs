# Resumo do dia - 22 de maio de 2026

## O que foi feito hoje

- **Migrações de base de dados com Alembic:** A aplicação passou a ter um sistema de versionamento de schema. Em vez de criar as tabelas diretamente ao arrancar, agora há um histórico de alterações que pode ser revertido — mais seguro e profissional.
- **Logs sem informação sensível:** Tokens de verificação de e-mail e de reset de palavra-passe deixaram de aparecer nos registos do servidor. Quem tiver acesso aos logs do Render não consegue ver esses tokens.
- **Redis obrigatório ao arrancar:** Se o sistema de limite de pedidos (Redis) não estiver acessível, a aplicação recusa-se a iniciar em vez de arrancar silenciosamente sem essa proteção.
- **Cabeçalhos de segurança melhorados:** A aplicação passou a enviar cabeçalhos HTTP que protegem os utilizadores — bloqueio de conteúdo de origens externas suspeitas (CSP), controlo de funcionalidades do browser (Permissions-Policy) e regras de cache que evitam guardar dados sensíveis no browser ou na CDN.
- **Backups inteligentes:** O script de backup automático deixou de ter uma lista fixa de tabelas. Agora descobre automaticamente todas as tabelas existentes — quando forem criadas tabelas novas (como a de edifícios), o backup incluí-las-á sem precisar de alteração manual.
- **Script de restore documentado:** Foi criado um procedimento completo para recuperar os dados da base de dados a partir de um backup, caso seja necessário em situação de emergência.
- **Guia de deploy centralizado:** A documentação de como colocar a aplicação em produção foi reorganizada e melhorada, incluindo uma secção sobre o segredo do GitHub necessário para o backup automático funcionar.
- **CORS mais restrito:** A aplicação em produção passou a validar que as origens dos pedidos HTTP são exclusivamente HTTPS e correspondem ao domínio oficial da aplicação.

## Estado do projeto

O ciclo de segurança que começou ontem avançou significativamente: de 4 planos concluídos passámos para 12 em 16. A aplicação está consideravelmente mais robusta — arranque seguro, logs limpos, headers de proteção, backups com restore documentado e migrações de schema versionadas. Restam 4 planos menores: validação de privilégios na base de dados, alertas de observabilidade, padronização de nomes de rotas API e documentação de deploy consolidada.
