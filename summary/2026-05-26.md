# Resumo do dia - 26 de maio de 2026

## O que foi feito hoje

- **Descoberto e corrigido o motivo pelo qual o login nunca funcionava em produção:** o Supabase tem uma funcionalidade de segurança ativa por defeito (Row Level Security) que tornava todos os registos da base de dados invisíveis para o utilizador da aplicação. O login retornava sempre erro 401 mesmo com a password correta, e os registos de actividade falhavam silenciosamente. O problema foi resolvido com duas linhas de SQL no painel do Supabase e uma migração automática para garantir que o fix se aplica em futuros re-deploys.

- **Corrigido um bug que bloqueava todos os formulários POST em produção:** o cookie de segurança CSRF não conseguia ser lido pelo frontend porque o backend e o frontend estão em domínios diferentes (`api.chichorrofireriskapp.joaopmteixeira.net` vs `chichorrofireriskapp.joaopmteixeira.net`). Corrigido com uma configuração de domínio no middleware. Confirmado a funcionar via inspeção dos headers HTTP.

- **Verificados todos os endpoints de saúde em produção:** `/health` e `/health/db` retornam os headers de segurança corretos, ID de pedido único e Cache-Control adequado.

- **Confirmado o sistema de backups automáticos:** o artifact gerado pelo GitHub Actions contém os três ficheiros esperados (`users.json`, `access_log.json`, `meta.json`). Os dados estavam a zero porque o RLS bloqueava as leituras — o próximo backup agendado (28 de maio) terá os dados reais.

- **Login confirmado a funcionar com o utilizador de produção:** após corrigir o RLS, o login com `JoaoTeixeira` e a password do 1Password funcionou sem problemas.
